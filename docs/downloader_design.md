# Concurrent Chunk Downloader: Architectural Contract

This document defines the interaction contract for the Chunk Downloader. It emphasizes **Architectural Governance** over implementation details.

## 1. The Core Philosophy: AI-Native Resilience
In an environment where code is generated by AI agents, the primary risk is **Non-Deterministic Error Handling**. AI often fails to anticipate complex failure modes of third-party APIs (e.g., HTTP 429 Rate Limiting vs. 503 Service Unavailable).

**AIX-Flow** shifts the responsibility of error handling from the AI-generated code to a **Human-Defined Atomic Policy**:
- **Isolated Side Effects**: In-flight I/O is confined to `.partial` staging files; only atomic renames publish results to their stable paths.
- **Deterministic Rollback**: Any failure triggers a mandatory, human-verified `Undo()`.
- **Manual Intervention Points**: Engineers define the "Critical Error Boundary" within the `Task` interface, the transport tuning on `DownloadTask.Client`, the budget on `BudgetedRunner`, and the AEAD invariants in the GCM writer.

## 3. Resume-Safe Chunk Write Contract

`DownloadTask` publishes bytes to `TempPath` via a **two-phase write**:

1. Stream the HTTP response body to `TempPath + ".partial"`.
2. Verify the downloaded length matches `Chunk.Size`.
3. `os.Rename(.partial, TempPath)` — atomic on POSIX.

**Invariants enforced by this contract**:

- Any file at `TempPath` is a complete, size-validated chunk. `GetPendingChunks` can treat its existence as authoritative proof of completion without reading or hashing the file.
- A crash anywhere before the rename leaves only a `.partial` file. The next run's `os.Create` truncates it and retries — no ambiguous state, no 0-byte-looks-completed trap.
- `Undo()` removes only `.partial`. A pre-existing `TempPath` from a previous successful attempt is sacred; `Undo` must not destroy neighbour chunks that happen to share a directory.

This decouples "bytes on disk" from "chunk committed to the merge set." The merge step and any resume logic both key off the published path, which only ever points at verified data.

## 4. Post-Processing Pipeline Contract
- **Compression**: Gzip, streamed via `gzip.Writer` so memory footprint is O(chunk size), not O(file size).
- **Encryption**: **AES-256-GCM** applied in a chunked streaming AEAD adapter (`gcmWriter`). GCM was chosen over CTR to provide **authenticated encryption** — tampering causes `aead.Open` to fail loudly, rather than silently producing corrupted plaintext.
- **Integrity**: SHA-256 computed over the final encrypted stream, exposed as `FinalHash` for transport-corruption detection. Authenticity is provided by the per-chunk GCM tags; the hash is a convenience fingerprint, not the security boundary.

### 4.1 On-Disk Format

```
[ 8 bytes ]   random salt
repeated until EOF:
  [ 4 bytes ] big-endian uint32 N          -- length of the sealed chunk
  [ N bytes ] ciphertext || 16-byte GCM tag
```

- Plaintext is split into fixed 64 KiB blocks before sealing. The final block may be shorter.
- Per-chunk nonce = `salt (8 bytes) || counter (4 bytes)`, where `counter` starts at 0 and increments once per sealed chunk.
- The salt is regenerated every `Execute()` from `crypto/rand`, ensuring `(key, nonce)` pairs are never reused across runs.

### 4.2 Engineer's Duties
- **AEAD Invariant**: Never reuse a `(key, nonce)` pair. If the per-chunk writer is cloned into another use site, the counter/salt construction must be preserved, or the 64 KiB chunking replaced with an equivalently unique nonce derivation.
- **Hash Scope**: `FinalHash` covers the entire on-disk stream (salt + all sealed chunks). Engineers changing the writer must verify this property manually — a regression that hashes only the ciphertext would silently shrink the detection window.
- **Budget Calibration**: When `PipelineTask` is wrapped in a `BudgetedRunner`, the timeout must accommodate worst-case compression ratios on large inputs. A premature cancel during `gzip.Close` or `gcmWriter.Close` skips the final sealed chunk, producing a truncated (but still authenticable) file that a naive consumer might accept.
